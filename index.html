LAB 1: Ubuntu Installation Documentation

Ubuntu Installation Using VMware

I began my Ubuntu installation journey by setting up a virtual environment using VMware Workstation. First, I downloaded the Ubuntu 22.04 LTS ISO file from the official Ubuntu website, ensuring I had the latest stable version for optimal security and performance.

I launched VMware Workstation and created a new virtual machine by selecting "Create a New Virtual Machine" from the home screen. I chose the "Custom (advanced)" option to have full control over the configuration settings. During the setup wizard, I allocated 4GB of RAM and 25GB of disk space to ensure smooth operation of the Ubuntu system.

I configured the virtual machine settings by specifying the downloaded Ubuntu ISO file as the installation media. I enabled virtualization features like VT-x/AMD-V to improve performance. After completing the initial setup, I powered on the virtual machine and was greeted with the Ubuntu installation screen.

I selected "Install Ubuntu" and proceeded through the installation wizard. I chose the default keyboard layout and connected to my network for updates during installation. For the installation type, I selected "Erase disk and install Ubuntu" since this was a virtual environment with no existing data to preserve.

I created my user account with a strong password and enabled automatic login for convenience in the lab environment. The installation process took approximately 20 minutes to complete. After the installation finished, I restarted the virtual machine and successfully booted into my new Ubuntu desktop environment.

Ubuntu Installation on Windows 11

For the Windows 11 dual-boot installation, I first ensured my system met the requirements and had secure boot enabled. I began by creating a backup of my important Windows files and created a recovery drive using Windows built-in tools.

I downloaded the Ubuntu 22.04 LTS ISO and created a bootable USB drive using Rufus. I configured Rufus to use GPT partition scheme for UEFI systems and selected the Ubuntu ISO file. The process took about 10 minutes to create the bootable media.

I accessed my Windows 11 disk management tool and shrunk my main partition to create unallocated space for Ubuntu. I allocated 50GB of free space, which I determined would be sufficient for the Ubuntu installation and my lab exercises.

I restarted my computer and accessed the BIOS/UEFI settings by pressing F2 during startup. I temporarily disabled secure boot and enabled legacy boot options to ensure compatibility with the Ubuntu installer. I changed the boot priority to boot from the USB drive first.

Upon booting from the USB drive, I selected "Try Ubuntu" to test the system before installation. After verifying everything worked correctly, I launched the installer from the desktop. I chose "Install Ubuntu alongside Windows Boot Manager" to maintain dual-boot capability.

The installer automatically detected my Windows installation and offered to install Ubuntu in the free space I had created. I proceeded with the automatic partitioning and completed the user account setup. After installation, I restarted and was presented with the GRUB bootloader, allowing me to choose between Windows 11 and Ubuntu at startup.

LAB 3: Testing Concurrency in Computing Systems

Understanding Concurrency in Practice

In my exploration of concurrency testing, I focused on understanding how modern computer systems handle multiple tasks simultaneously. Concurrency is the foundation of multitasking operating systems, allowing me to run multiple applications while maintaining system responsiveness.

I began my testing by creating multiple concurrent processes and observing how the Linux kernel's scheduler managed them. Using the stress utility, I generated artificial CPU, memory, and I/O loads to simulate real-world concurrent scenarios.

CPU Concurrency Testing

I installed the stress testing tool and ran multiple CPU-intensive tasks simultaneously. Using the command stress --cpu 8 --timeout 60s, I created 8 worker threads that maxed out my CPU cores. Through htop monitoring, I observed how the system maintained responsiveness despite high CPU utilization.

# Installing stress testing tools $ sudo apt install stress stress-ng # Running concurrent CPU tests $ stress --cpu 4 --vm 2 --vm-bytes 1G --timeout 30s

I monitored the results using multiple terminal windows running different monitoring tools simultaneously. This practical demonstration of concurrency showed me how I could run htop, vmstat, and iotop all at the same time while the stress test was executing.

Memory Concurrency Challenges

Memory concurrency presented unique challenges that I explored through practical testing. I created scenarios where multiple processes competed for memory resources using stress --vm 4 --vm-bytes 512M. This command spawned 4 processes, each allocating and deallocating 512MB of memory continuously.

I observed how the Linux kernel's memory management system handled concurrent memory requests. Using the free -h command in a watch loop, I monitored real-time memory usage and saw how swap space was utilized when physical memory became scarce.

I/O Concurrency Testing

File system concurrency testing revealed interesting insights about disk performance under concurrent loads. I used stress --io 4 to create multiple processes performing continuous read/write operations. Simultaneously, I monitored disk activity using iotop and iostat.

I discovered that modern SSDs handle concurrent I/O operations much more efficiently than traditional hard drives. My testing showed consistent performance even with multiple processes accessing the disk simultaneously, demonstrating the effectiveness of modern storage controllers and filesystems.

Network Concurrency Experiments

To test network concurrency, I set up multiple simultaneous connections using tools like wget and curl. I downloaded multiple files concurrently and monitored network usage with iftop and nethogs.

I observed how the TCP/IP stack managed multiple concurrent connections and how bandwidth was shared among different processes. This testing helped me understand network socket management and connection pooling concepts in practice.

Process Synchronization Observations

Through my concurrency testing, I gained practical insights into process synchronization mechanisms. I observed how the kernel prevented race conditions and managed shared resources. Using tools like ps aux and pstree, I could visualize process relationships and understand parent-child process hierarchies.

I experimented with process priorities using nice and renice commands, learning how I could influence the scheduler's decision-making process. This hands-on experience demonstrated the importance of proper resource management in concurrent computing environments.

Real-World Concurrency Applications

My testing concluded with practical applications of concurrency principles. I set up scenarios mimicking real-world server environments where multiple users access services simultaneously. Using simple Python scripts, I created multiple client connections to test how services handle concurrent requests.

This comprehensive testing approach gave me valuable insights into system behavior under concurrent loads and helped me understand the critical role of concurrency in modern computing systems. The experience reinforced the importance of proper system monitoring and resource management in cybersecurity contexts.
